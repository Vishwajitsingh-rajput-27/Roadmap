4.1 â€” How the Web Actually Works
HTTPClient-ServerRESTStatus Codes
â–¼

Before you write backend code, understand the system you're building inside.

The Client-Server Model

Client = your browser (the frontend). Server = a computer somewhere that stores data and does logic (the backend). When you type a URL, here's exactly what happens:

1. You type "www.netflix.com" â†’ your browser (client) sends a REQUEST
2. The request travels over the internet to Netflix's SERVER
3. The server processes it â€” checks who you are, gets your shows, etc.
4. The server sends back a RESPONSE (the HTML/data)
5. Your browser displays it on screen

This whole thing takes ~0.3 seconds. Every page load does this.
HTTP Methods â€” The Verbs of the Web
GET    /users          â†’ "Give me all users"       (read)
POST   /users          â†’ "Create a new user"       (create)
PUT    /users/1        â†’ "Update user #1"          (update)
DELETE /users/1        â†’ "Delete user #1"          (delete)

These 4 together = CRUD (Create, Read, Update, Delete)
Every app you've ever used is built on CRUD.
HTTP Status Codes â€” What do they mean?
200 OK              â†’ Success! Here's what you asked for.
201 Created         â†’ Success! New thing was created.
400 Bad Request     â†’ You sent something wrong.
401 Unauthorized    â†’ You're not logged in.
403 Forbidden       â†’ You don't have permission.
404 Not Found       â†’ That thing doesn't exist.
500 Server Error    â†’ Something broke on our end.
What is a REST API?

REST is just a set of rules for how client and server talk. A REST API uses HTTP methods + URLs to do things. Example: GET /api/products returns all products. POST /api/products creates one. The /api/ prefix tells you it's an API, not a webpage. You'll build one of these.

4.3 â€” Authentication: Login & Signup
bcryptJWTLogin FlowProtected Routes
â–¼

Authentication = proving who you are. This is how your website knows "this is Alex" vs "this is someone else." It's one of the most critical skills for any full-stack app.

The Login Flow â€” Step by Step
1. User enters email + password on login page
2. Frontend sends POST /api/auth/login  { email, password }
3. Server looks up the user by email in the database
4. Server compares the sent password to the stored hash (using bcrypt)
5. If they match â†’ server creates a JWT token and sends it back
6. Frontend stores the token (in memory â€” NOT localStorage ideally)
7. On every future request, frontend sends the token in the header:
      Authorization: Bearer <token>
8. Server checks the token â†’ if valid, the request continues
Password Hashing with bcrypt
const bcrypt = require('bcryptjs');   // npm install bcryptjs

// â”€â”€ When user SIGNS UP: â”€â”€
async function signup(email, password) {
  const salt   = await bcrypt.genSalt(10);          // creates random salt
  const hashed = await bcrypt.hash(password, salt); // scrambles password
  // Store 'hashed' in database â€” NEVER store plain password!
  console.log(hashed);  // "$2a$10$Xk9..." â€” unreadable = safe
}

// â”€â”€ When user LOGS IN: â”€â”€
async function login(enteredPassword, storedHash) {
  const match = await bcrypt.compare(enteredPassword, storedHash);
  // match = true or false
  if (match) console.log("Password correct!");
  else       console.log("Wrong password");
}
JWT Tokens â€” Proof of Identity
const jwt = require('jsonwebtoken');  // npm install jsonwebtoken

// â”€â”€ CREATE a token (after successful login) â”€â”€

// â”€â”€ VERIFY a token (middleware for protected routes) â”€â”€
function authenticate(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1];  // "Bearer xxx" â†’ "xxx"
  if (!token) return res.status(401).json({ msg: 'No token' });

  try {
    const decoded = jwt.verify(token, SECRET);
    req.user = decoded;   // attach user info to the request
    next();               // continue to the next route
  } catch (err) {
    res.status(401).json({ msg: 'Invalid token' });
  }
}
